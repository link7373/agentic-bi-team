# Build Dashboard

Owner: `dashboard-developer`, with `analytics-engineer` for the data layer. Args describe the dashboard, e.g. `/build-dashboard executive revenue overview`.

## Procedure

1. **Challenge the request (briefly).** Is this a recurring monitoring need (dashboard) or a one-off question (`/analyze` is cheaper)? Does an existing dashboard in `dashboards/README.md` already cover 80% of it? Proceed once it's genuinely a dashboard need.

2. **Requirements.** Identify: audience (from `knowledge/stakeholders.md`), the 1–3 questions it must answer, metrics (must exist in `knowledge/metrics-catalog.md` — route new ones through `/define-kpis`), filters/drill paths, refresh cadence, access level.

3. **Spec.** Write `dashboards/<name>/SPEC.md`: all of the above plus a text/ASCII layout sketch following `standards/dashboard-standards.md` (headline number top-left, supporting trends, detail below the fold). Pick each chart from the selection/charts-to-avoid guidance there (length/position over area; no pies for comparison; colour-blind-safe; bars from zero — fuller rationale in `analytics.md` Part 2). For stakeholder-facing dashboards, confirm the spec with the user before building.

4. **Data layer.** Every tile reads from a mart or summary table — no heavy logic in the BI tool. Missing tables → build via `/build-pipeline` first. Confirm the source tables refresh at least as often as the dashboard promises.

5. **Build in {{BI_TOOL}}.**
   - *Tableau:* published data source + workbook; extracts for slow sources; calculated fields documented in the spec.
   - *Power BI:* star-schema model, thin DAX measures named per catalog, dedicated date table.
   - *Looker:* LookML in `dashboards/lookml/`, measures defined once at model layer, business-language labels.
   - *No BI tool configured:* deliver a self-contained HTML dashboard (vanilla JS + a charting lib, or static matplotlib images) regenerated by a script in `dashboards/<name>/`.
   - If direct tool access (API/MCP/CLI) isn't available, produce the complete artifact that can be imported (e.g. .twbx instructions, .pbit steps, LookML files) plus exact setup instructions.

6. **Validate.** Reconcile every displayed number against an independent direct query (save reconciliation queries in `dashboards/<name>/checks/`). Test filters, empty states, and the slowest interaction. Five-second test on the layout.

7. **Ship.** Screenshot to `dashboards/<name>/`, update `dashboards/README.md` inventory (name, audience, source tables, refresh, owner, link). Publishing to a broad or external audience → confirm with the user first.

8. **Maintain.** Note in the spec when the dashboard should be reviewed for retirement ({{DASHBOARD_REVIEW_CADENCE e.g. quarterly}}). Dead dashboards erode trust in all dashboards.
